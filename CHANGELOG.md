# Changelog

All notable changes to this project are documented here. The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html) at the system level.

Each `SKILL.md` carries a `version:` field in its frontmatter showing the system version it last meaningfully changed in. To check what's installed: `grep -E "^name:|^version:" ~/.claude/skills/*/SKILL.md`.

## [Unreleased]

Third-wave retro-driven rule additions — the first produced by `/retro-mine` (17 retros, 2026-05-30 → 2026-07-15). Four rule-not-working strengthenings (verify-premises generalized to all claims, Bash loophole closed in the orchestrator edit hook, merge ask-gate as structural backstop for inferred merge authority, live-surface verification extended to UI work) and nine new rules, each with ≥2-retro evidence.

### Added
- **`_shared/orchestration.md` §"Claims are hypotheses until verified"** — verify-premises generalized beyond bd/git state: severity verdicts, proof-of-fix claims, mitigation applicability, and numeric counts must be source/test-verified in-session or labeled unverified; persona verdicts are never evidence (6 retros).
- **`_shared/orchestration.md` §"Open decisions block dependent dispatch"** — "queued pending a PO decision" is a hard dispatch block; flagged design-forks block their dependent build steps; catch-alls don't resolve forks (5 retros).
- **`_shared/orchestration.md` §"Irreversible follow-on actions state their authorization inline"** + `_shared/decision-prompts.md` — decision options that imply a later merge/deploy/delete/publish state that action's authorization status in the option text; one-way outward-facing actions require enumerative review or explicit PO sign-off on sampling (4 retros).
- **`_shared/orchestration.md` §"Data-Integrity Changes Get the DBA"** + `database-engineer/SKILL.md` — schema/query/transaction/bulk-write changes get the DBA as a required additional reviewer; severity requires tracing the full failure-propagation path; data-integrity kickback fixes get a full second review round (2 retros).
- **`hooks/pretooluse.py` Rule A2** — orchestrator Bash-mutation block: `sed -i`/`perl -i`/`patch` and redirect/`tee` into project-tree paths denied for the main agent in onboarded projects (the Bash loophole that let hand-edited CSS ship 5 days after Rule A landed). Self-check grown to cover it. Plus two new named rationalizations ("just a dep bump," "just CSS polish").
- **`install.sh` merge ask-gate** — installer adds `ask`-tier permissions on `Bash(gh pr merge*)` / `Bash(git merge*)` to settings.json (global by PO decision): merges always require a live human click, killing authorization-by-momentum mechanically.
- **`_shared/engineering-discipline.md`** — five new sections: UI work renders before it's done (render-before-done gate, category-spanning smokes); confirm the deployed artifact before hypothesizing a regression; real fixtures before third-party integration code; single-occupant verification environments; enforcement code tests itself (fixture self-test + dogfood, same session); background processes stay observable (file+heartbeat, `ps` on stall, second-identical-failure ripcord).
- **`project-engineer/SKILL.md` §"Third-Party Data: Real Fixtures First"** — field-value survey pasted into the bead before writing filters/matchers/enums; ≥1 test parses a recorded real response (2 retros).
- **`qa-engineer/SKILL.md` §"Tests Must Be Able to Fail"** — name-matches-behavior check, untested critical paths are blocking by default, regression fixes extract testable pure functions, third-party-data acceptance criteria, flakiness sweeps before orchestrator verification (4 retros).
- **`code-reviewer/SKILL.md` §"Ship Checklist Is a Review Item"** — shipping-process adherence as a first-class review check; duplicated-logic fixes land at every site; enforcement briefs get design review; review doesn't launder unverified premises (3 retros).
- **`release-check/SKILL.md`** — ship-checklist rule: version advanced (not merely consistent), CHANGELOG rides the same PR, documented local checkers run.
- **`technical-writer/SKILL.md` §"Docs Are Definition-of-Done, Not Follow-Up"** + `project-manager/SKILL.md` §"Docs Beads Are Definition-of-Done" — operator runbooks ship with destructive features, CHANGELOG-in-PR, every copy surface owned, docs beads filed at scoping time; follow-ups never parent under closed epics (4 retros).
- **`retro-mine/LAST_MINED`** advanced to `2026-07-15-14_retro-pipeline-buildout.md`. Watch items carried in the mining report only (1 retro each): `pkill -f` self-matching, repo-observed PR-workflow vs CLAUDE.md conflicts, haiku-downshift exemption for code-adjacent reviewers, bd version pinning, `.beads/issues.jsonl` merge driver, EXPLAIN-at-scale, marathon checkpointing, GH-issue intake scans.

Second-wave retro-driven rule additions (7 retros, 2026-05-17 → 2026-05-29). Tightens reviewer sub-agent discipline, introduces verification-against-the-live-surface as a tier-invariant rule, codifies "no orchestrator-finish" on mid-run termination, and adds a recurring-regression test-strategy pattern.

### Added
- **`retro-sync/` skill (SKILL.md + scrub.sh)** — retros now sync to a single public source of truth ([EndofLineTech/retros](https://github.com/EndofLineTech/retros)) instead of per-machine `~/retros/` dirs shuttled by SCP. One `/retro-sync` run pushes local retros and pulls everyone else's; contributors without write access fork + PR via `gh`. Anonymization is two-layer: generation-time rules in `retro/SKILL.md` (new §Anonymization — project pseudonyms, no credentials/PII/hostnames, describe process not payloads) and a deterministic `scrub.sh` backstop (stable project-name pseudonymization from a private local map `~/.claude/retro-project-map.txt` applied to content and filenames, plus redaction of emails, IPs, private keys, AWS/GitHub/Anthropic/Slack/bearer credentials, and literal denylist terms from `~/.claude/retro-scrub.txt`). `/retro` gains a final step: run `/retro-sync`. Registered in both installers.
- **`retro-mine/` skill (SKILL.md + LAST_MINED watermark)** — formalizes the retro-driven rule-addition passes previously done by hand. `/retro-mine` syncs the corpus, mines retros newer than the watermark via sonnet reader agents (~5 retros each, quote-level evidence required), clusters findings at the orchestrator (≥2 retros = rule candidate, 1 = watch item), detects rule-not-working signals (pattern recurring in retros dated after the covering rule landed → propose strengthening/enforcement, not a duplicate rule), and presents a `## DECISIONS NEEDED` block — no skill file is edited before PO approval. Watermark seeded at the last manual pass (2026-05-29). Registered in both installers and the orchestration model table.
- **`code-reviewer/SKILL.md` §"Tool Discipline When Spawned as Sub-Agent"** — read-only effective tool set when dispatched as a review sub-agent. Explicitly forbids `Edit`/`Write`/`NotebookEdit` and state-mutating Bash regardless of inherited tool access. Brief-level instruction is the fence.
- **`code-reviewer/SKILL.md` §"PR Review Summary"** — verdict header must match body severity. `Approved`-class headers reserved for non-blocking observations; any blocker in the body requires `Changes Requested` or `Blocked`. Downstream readers and orchestrators key on the header for merge decisions.
- **`qa-engineer/SKILL.md` §"When Spawned as a Review Sub-Agent"** — combined tool discipline + verdict-header rule for QA when invoked in review mode (`/team-review`, `/release-check`, or direct dispatch).
- **`qa-engineer/SKILL.md` §"Recurring Regressions Are a Test-Strategy Signal"** — when the same surface regresses across 3+ rounds with unit tests green, the missing layer is integration-with-real-data. Golden-fixture end-to-end tests and parity/boundary tests as the response, not more unit tests.
- **`_shared/engineering-discipline.md` §"Persisted and Live-Surface Verification"** (subsection of Verification of Completion) — for reporting/integration bugs, inspect the persisted artifact and trace `compute → persist → serialize → format` before editing the compute layer. Synthetic verification (hand-built inputs to one function) produces false confidence; live verification requires the real surface.
- **`_shared/engineering-discipline.md` §"Stage Unvalidated Fixes"** — when a fix is speculative and shipped alongside instrumentation, ship instrumentation alone first so the next data round can isolate variables. Bead state for unvalidated fixes must be distinguishable from closed-and-proven.
- **`_shared/orchestration.md` §"Mid-run termination is not orchestrator-finish authorization"** (subsection of Claude Orchestrates — Personas Implement) — when a persona sub-agent terminates mid-run with uncommitted work, the orchestrator re-dispatches the persona; it does NOT commit, push, or finalize the persona's pending work itself.
- **`_shared/orchestration.md` §"Reviewer briefs require explicit tool discipline"** (subsection of Worktree Isolation) — verbatim brief language requiring read-only operations for any persona dispatched in review mode. Pair to the `code-reviewer`/`qa-engineer` SKILL-level discipline.
- **`install.sh --project <dir>` (+ `--local`) and `uninstall.sh --project <dir>`** — project-scoped installation for users who don't want the team in their entire Claude workflow. Installs skills/agents/hook/CLAUDE.md block under the target project's `.claude/` and `CLAUDE.md`; forces copy mode (symlinks into a personal clone break for anyone else who clones the project), making the install committable — teammates get the whole team via git. Hook command uses `$CLAUDE_PROJECT_DIR` so no user-specific path is committed. `--local` registers the hook in `settings.local.json` and prints `.gitignore` guidance for a personal-only install. Project uninstall removes the hook script copy, tidies empty `.claude/` husks, and cleans both settings variants while preserving unrelated content. `~/retros` remains the one global surface (retros span projects). Sandbox-tested: global regression, double-install idempotency in both scopes, `--local`, error cases (`--local` without `--project`, `--project` pointed at this repo), and uninstall symmetry.
- **`hooks/pretooluse.py` + install.sh/uninstall.sh settings.json management** — PreToolUse enforcement dispatcher converting the mechanically decidable orchestration rules from prose to deterministic denials: (A) orchestrator edit block in onboarded projects (main agent — no `agent_id` in hook input — may not Edit/Write project files; `COMPONENTS.md`/`CLAUDE.md`/`.claude/` and out-of-tree paths exempt as orchestrator territory), (1) ceremony gate (the six `COMPONENTS.md`-requiring ceremonies deny at the `Skill` tool call without it), (2) persona bead firewall (subagents denied `bd create/close/delete/reopen` per "Findings Are Notes, Not Beads" and board-state-is-orchestrator-territory; `bd update` and `beads:*` agent types exempt), (3) bead-referenced commits (`git commit -m` in `.beads/` repos requires a bead id; `[no-bead]` escape hatch). Deny reasons double as just-in-time orchestration reminders, immune to context drift. The claude-agent-dev-team repo itself is exempt (meta-work). install.sh registers the hook in `~/.claude/settings.json` idempotently (preserves unrelated hooks/settings); uninstall.sh removes only our entry. Stdlib-only Python; self-check via `python3 hooks/pretooluse.py --check` (17 assertions). Not yet wired into install.ps1 (bash/python3 assumption).
- **README §"Recommended: beads"** — explicit upfront recommendation to install [beads](https://github.com/steveyegge/beads) as the work board, with the honest trade-off (skills fall back to conversation context without it; nothing persists between sessions) and a pointer to the board-tool swap in Customization. Beads was already the assumed default across all ceremonies but was only mentioned in passing.
- **`_shared/claude-md-project-template.md`** — recommended project-level `CLAUDE.md` block for repos run by the team: tiers pointer, board, gates, branching, deploy, off-limits areas. Per-project facts only — orchestration discipline stays in the installer-managed global block. `/onboard` Step 3e (new) offers to write it after `COMPONENTS.md`, filled from what onboarding observed, inside managed markers so re-onboarding replaces only the block.

### Added
- **`agents/persona-reviewer.md`** — read-only custom agent type for review-mode persona dispatch (no `Edit`/`Write`/`NotebookEdit`; the persona still comes from the prompt). Makes the worst class of reviewer worktree corruption structurally impossible instead of prose-forbidden. Installed to `~/.claude/agents/` by install.sh; `/team-review`, `/team-plan`, `/grooming`, `/standup`, and orchestration.md now prefer it with `general-purpose` as fallback (those ceremonies' spawns are all read-only — grooming's bead updates happen at the orchestrator level). `/spike`, `/postmortem`, and `/onboard` stay on `general-purpose`: spikes may build proof-of-concepts, and the others weren't in the validated scope. CONTRIBUTING.md documents the `agents/` directory. The brief-level fence text remains required — it's the only guard on destructive Bash. This is also the validation beachhead for a possible future conversion of personas to native agent definitions.
- **`_shared/decision-prompts.md`** — full `## DECISIONS NEEDED` format reference (example block, dependency ordering, one-by-one mode, anti-patterns), extracted from orchestration.md. Read on first decision block of a session instead of on every session start.
- **`release-check` Step 5 — version/changelog consistency** — before tagging, verify artifacts listed under `[Unreleased]` carry the new release version (SKILL.md `version:` frontmatter in skill repos, manifest versions in code repos). New checklist row; subsequent steps renumbered.

### Changed
- **`_shared/orchestration.md` slimmed for every-session load** (it's read at the start of nearly every session via the managed CLAUDE.md block): §"Decision Prompts" reduced to the rule + pointer at `_shared/decision-prompts.md`; §"Review History Before Re-Reviewing" reduced to the rule + pointer at `code-reviewer/SKILL.md` §"Review History Discipline" (which was a near-verbatim duplicate). Net ~425 words off every session start (3,804 → 3,379, after adding the structural-enforcement guidance below); no rules removed.
- **`sre/SKILL.md` `when_to_use` trimmed** from 19 trigger phrases to 12 — the dropped tail (metrics design, logging architecture, distributed tracing, dashboard design, alert tuning, observability cost management, observability pipeline) is subsumed by observability/monitoring/instrumentation. Skill descriptions load into every session's context, so trigger-list length is a universal cost.
- **`_shared/orchestration.md` §"Verify Premises Before Briefing"** — added two data-check bullets: post-disconnect / post-gap re-verify (re-query board and PR state after any context gap before forwarding to the next agent) and HEAD-over-working-tree (verify briefs that assert "engineer's pushed code does/doesn't X" against `git show HEAD:<file>`, not the working tree).
- **Token-efficiency pass across ceremonies and personas** (audit 2026-07-06):
  - **`/team-plan` + `/team-review` quick mode now reads `identity.md` only** — quick-mode persona agents no longer load full SKILL.md + engineering-discipline.md + conflict-resolution.md (~10k tokens/agent, ~100k/run across 10 agents); identity.md (~270 tokens) carries what triage-depth analysis needs. Full mode unchanged. Pattern copied from `/standup` Phase 1.
  - **`/grooming` defaults to `identity.md`** — sizing and acceptance criteria are pattern-matching. New **deep-groom** escape hatch (PO-requested, or beads flagged complex at the value gate) reads full SKILL.md for the relevant personas only.
  - **Reviewer tool-discipline sections deduplicated at runtime** — `code-reviewer` §"Tool Discipline When Spawned as Sub-Agent" and `qa-engineer` §"When Spawned as a Review Sub-Agent" no longer restate the forbidden-operation list verbatim; the dispatching brief already carries the binding fence per `_shared/orchestration.md` §"Reviewer briefs require explicit tool discipline". The SKILL sections keep the rule ("READ-ONLY; report, don't fix; follow the fence even if the brief omits it") without the double-delivered enumeration.
  - **Tier-calibration preambles slimmed** in `sre`, `qa-engineer`, `technical-writer`, `database-engineer`, `it-architect` — per-tier example enumerations removed (they duplicate the per-persona calibration tables in `_shared/deployment-tier.md`, which the preamble now points at explicitly). Persona-unique rules retained (DBA tier-invariance of data integrity, architect "Phase 1 forever is correct").

## [0.3.0]

Orchestration discipline upgrades from retro analysis (18 sessions, 2026-04-19 → 2026-05-14). New `/release-check` skill, expanded orchestrator rules, and a pre-existing-failure escalation clause.

### Added
- **`/release-check` skill** — pre-release semantic readiness check. Verifies open P0/P1 bugs in sprint scope, in-flight verification agents, open PRs targeting the release branch, and security gate state. Produces a checklist for PO confirmation and refuses to fire release-execution without explicit go.
- **`_shared/orchestration.md` §"Authorization Verbs"** — verb-mapping table ("ship it" → push+PR, "merge it" → merge, "go" → execute the immediately-prior proposed action) and rule for catch-all instructions vs flagged decisions.
- **`_shared/orchestration.md` §"Findings From Personas Are Notes, Not Beads"** — orchestrator-level rule against filing beads on persona-surfaced sibling concerns without PO authorization.
- **`_shared/orchestration.md` §"Re-Verify Gates on Engineer Report"** — engineer's "gates green" report is the start of verification, not the end; orchestrator re-runs gates against the merged or worktree state before declaring done.
- **`_shared/orchestration.md` §"Definition of Done for User-Reported Bugs"** — merged ≠ deployed ≠ reporter notified ≠ reporter confirmed.

### Changed
- **`_shared/orchestration.md` §"Claude Orchestrates — Personas Implement"** — rewritten around "skills define each persona's scope." Adds hard limit of 3 pre-dispatch reads, explicit orchestrator-territory vs persona-territory enumeration (durable artifacts are always persona work, including bead descriptions, ADRs, in-repo docs), named-rationalization rejections, and a mid-task drift clause.
- **`_shared/orchestration.md` §"Worktree Isolation for Write-Mode Agents"** — trigger expanded from "will commit" to "might commit." New subsections: orchestrator-self discipline (verify `pwd`/`branch`/`status` before destructive git ops; delegate cleanup to engineer when parallel agents are running) and CWD drift (prefer absolute paths over `cd` chains).
- **`_shared/orchestration.md` §"Verify Premises Before Briefing"** — added framing checks (source/environment, already-shipped, inherited premise) alongside the existing data checks (pagination, child counts, status). Default assumption for user reports is now "customer in production."
- **`_shared/orchestration.md` §"Decision Prompts"** (renamed from "Decision Prompt Compression") — structured `## DECISIONS NEEDED` block at end of message, dependency notation, hard cap of 3 decisions per message, one-by-one mode for complex decisions.
- **`_shared/engineering-discipline.md` §"Pre-Existing Failures Are Not Background Noise"** — added escalation clause: pre-existing failures recurring across sessions escalate to RED at the next standup.

## [0.2.0]

Tier-aware personas, per-task model selection, and explicit versioning.

### Added
- `_shared/deployment-tier.md` — defines four deployment tiers (`home-lab`, `small-team`, `startup`, `enterprise`) and per-persona calibration tables. Personas read this and the project's `COMPONENTS.md` to right-size recommendations to the deployment context.
- `COMPONENTS.md` (per-project artifact) produced by `/onboard`. Lists each component in the project with its deployment tier. Required for the team ceremonies to run.
- New Step 3 in `/onboard` — Component Inventory & Tier Confirmation. The IT Architect drafts a component list with proposed tiers; other personas contribute tier signals; the PO confirms; `COMPONENTS.md` is written.
- **Preflight check** in `/team-plan`, `/team-review`, `/standup`, `/grooming`, `/spike`, `/postmortem` — refuse to run if `COMPONENTS.md` is missing, directing the user to `/onboard`.
- **Cross-tier resolution rule** — when work spans components at different tiers, strictest tier wins by default. Surface as a decision when applying it across the board would be clearly wasteful.
- **Agent Model Selection** section in `_shared/orchestration.md` — per-task-type model assignments (`haiku` for triage, `sonnet` for most domain work, `opus` for sticky decisions). Tier modulation: home-lab effective tier downshifts one level *except* for security-engineer.
- `model:` field in every persona `SKILL.md` frontmatter (default `sonnet`) — declares the persona's default when invoked directly.
- `version:` field in every `SKILL.md` frontmatter — echoes the system version the skill last changed in.
- README sections: "Project Onboarding (Required Before Team Skills Run)", "Tier-Aware Personas", "Per-Task Model Selection", "Versioning & Rollback".
- `CHANGELOG.md` (this file).
- `CONTRIBUTING.md` updates: documented `model:` and `version:` frontmatter, semver policy, and per-skill version bump expectations.

### Changed
- Persona "Hard Rules" sections in `sre/`, `technical-writer/`, `qa-engineer/`, `database-engineer/`, `it-architect/` reframed from absolute (`No exceptions`) to tier-conditional. Maximum-rigor wording preserved as the *enterprise-tier* baseline; lower tiers documented per-rule.
- Persona preambles (sre, technical-writer, qa-engineer, database-engineer, it-architect) now reference `_shared/deployment-tier.md` and instruct the persona to read `COMPONENTS.md` before producing recommendations.
- `/standup` agent prompts now instruct personas to rate R/Y/G against each component's own tier — a home-lab service is not RED for missing enterprise practices.
- Single-persona invocations (`/sre`, etc.) ask which tier the work is for if `COMPONENTS.md` doesn't cover the in-scope component.

### Tier-Invariant Behavior (Unchanged)
- Critical security findings (CVSS 9.0+) remain non-negotiable at every tier. The CVE in a home-lab service is still a CVE.
- Engineering Discipline (`_shared/engineering-discipline.md`), the Conflict Resolution Protocol, naming discipline, the value gate, and AI-realistic effort estimates apply at every tier.
- Vendor portability and open-source-tooling rules in `it-architect/` remain tier-invariant.
